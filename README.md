# TraderBot — Спекулятивная торговля на MOEX

Автоматизированный торговый бот для российского фондового рынка через **T-Bank (Tinkoff) Invest API**.
Бот получает свечные данные, ищет сетапы по стратегии ICT, выставляет лимитные ордера со стоп-лоссом и тейк-профитом прямо на бирже, а также поддерживает бэктест на исторических данных.

```bash
pip install -r requirements.txt
```

### 4. Настройка токена

Создайте файл `.env` в корне проекта:

```bash
# Windows
copy .env.example .env

# Linux / macOS
cp .env.example .env
```

Откройте `.env` и вставьте свой токен:

```env
TINKOFF_TOKEN=your_token_here

```

Токен можно получить в личном кабинете **T-Bank Инвестиции**:
`Настройки → Токен для Tinkoff Invest API → Создать токен`

> Чтобы переключиться с песочницы на реальную торговлю, измените `SANDBOX=false`.

### 5. Запуск

```bash

py -3.12 main.py

# Бэктест
py -3.12 backtest.py

# Тесты
py -3.12 -m pytest tests/ -v
```

## Стратегия — ICT System Variant A

Стратегия основана на концепции ICT (Inner Circle Trader): поиск сбора ликвидности (sweep) на старшем таймфрейме и вход на младшем через импульсную свечу (displacement).

### Шаг 1 — Поиск sweep на 1H

Анализируется окно из **15 последних закрытых свечей** на часовом графике (structure window). Текущая (последняя) свеча проверяется на пробой этого окна:

**Бычий sweep (сигнал BUY):**
```
low текущей свечи < min(low последних 15 свечей)   — фитиль пробил структурный минимум
AND
close текущей свечи > структурный минимум            — свеча закрылась обратно выше
```
Интерпретация: маркет-мейкеры собрали стоп-лоссы покупателей ниже уровня, после чего цена вернулась — ложный пробой вниз.

**Медвежий sweep (сигнал SELL):**
```
high текущей свечи > max(high последних 15 свечей)  — фитиль пробил структурный максимум
AND
close текущей свечи < структурный максимум           — свеча закрылась обратно ниже
```
Интерпретация: маркет-мейкеры собрали стоп-лоссы продавцов выше уровня, после чего цена вернулась — ложный пробой вверх.

Пробитый уровень (`sweep_level`) запоминается — он используется для расчёта стоп-лосса.

### Шаг 2 — Поиск displacement на 30m

Если sweep подтверждён, бот ищет **импульсную свечу** (displacement) на 30-минутном графике. Рассматриваются только свечи, открывшиеся **после начала sweep-свечи** на 1H.

Условия displacement-свечи:
```
body = abs(close - open)
range = high - low
body / range >= 0.35  (тело >= 35% всего диапазона свечи)
```

Дополнительно, свеча должна идти **в направлении сделки**:
- BUY: `close > open` (бычья свеча)
- SELL: `close < open` (медвежья свеча)

Берётся **первая** подходящая displacement-свеча.

### Шаг 3 — Расчёт уровней входа

**Entry (вход)** — лимитный ордер на **50% ретрейсмента тела** displacement-свечи:
```
BUY:  entry = close - 0.50 * body
SELL: entry = close + 0.50 * body
```

**Stop-loss** — на 0.3% за sweep-уровнем:
```
BUY:  stop = sweep_level * (1 - 0.003)    — ниже структурного минимума
SELL: stop = sweep_level * (1 + 0.003)    — выше структурного максимума
```

**Take-profit** — по заданному Risk/Reward:
```
risk = abs(entry - stop)

BUY:  target = entry + RR * risk
SELL: target = entry - RR * risk
```

RR = 2.0 (оптимальное значение по результатам бэктеста).

### Визуальная схема (BUY-сетап)

```
Цена
  ^
  |
  |                         TP (entry + RR * risk)
  |                          ────────────────
  |
  |                         Entry (50% ретрейсмент displacement)
  |                          ────────────────
  |
  |   ┌──── Structure low (min за 15 свечей)
  |   │      ────────────────
  |   │
  |   └──── Stop-loss (sweep_level * 0.999)
  |          ────────────────
  |
  └──── Sweep (фитиль ниже structure low)
```

## Риск-менеджмент

Модуль `RiskManager` рассчитывает размер позиции по правилу **1% риска на сделку**:

```
risk_amount = balance * 0.01              — максимальный убыток в рублях
risk_per_unit = abs(entry - stop)         — убыток на 1 акцию при стопе
qty = floor(risk_amount / risk_per_unit)  — количество лотов (минимум 1)
```

Пример: баланс 100 000 RUB, entry=300, stop=297 → risk_amount=1000, risk_per_unit=3 → qty=333 акции.

## Исполнение ордеров (ExecutionManager)

### Открытие позиции

1. Запрос баланса портфеля через API
2. Расчёт размера позиции через `RiskManager`
3. Размещение **лимитного ордера** на вход по цене `entry_price`
4. **Сразу** размещение стоп-лосса на бирже (`STOP_ORDER_TYPE_STOP_LOSS`)
5. **Сразу** размещение тейк-профита на бирже (`STOP_ORDER_TYPE_TAKE_PROFIT`)

SL и TP живут на бирже — позиция защищена, даже если бот упадёт.

### Закрытие по таймауту

Каждые 30 минут бот вызывает `update()` для каждой открытой позиции. Если позиция прожила **20 свечей по 30m (10 часов)**, бот:
1. Отменяет лимитный ордер на вход (если ещё не исполнен)
2. Отменяет стоп-лосс и тейк-профит на бирже
3. Закрывает позицию по рыночной цене
4. Записывает результат в журнал

### Типы закрытия

| Причина        | Описание                              |
|----------------|---------------------------------------|
| `take_profit`  | Цена достигла TP — закрытие биржей    |
| `stop_loss`    | Цена достигла SL — закрытие биржей    |
| `timeout`      | 20 свечей 30m прошло — закрытие ботом |

```
Запуск
  │
  ├── Загрузка токена из .env
  ├── Инициализация TinkoffBroker (sandbox или live)
  ├── Если sandbox: создание аккаунта + пополнение 100 000 RUB
  ├── Инициализация Storage, Strategy, Risk, Journal, Execution
  │
  └── Бесконечный цикл (каждые 60 секунд):
        │
        └── Для каждого тикера (6 штук):
              │
              ├── Получение 1m свечей за 3 дня
              ├── Ресемплирование в 30m и 1H
              ├── Обновление открытых позиций (проверка таймаута)
              │
              └── Если нет открытой позиции:
                    ├── Поиск сетапа через ICTStrategy
                    └── Если сетап найден → открытие позиции
```

## Бэктест (backtest.py)

Walk-forward симуляция на исторических данных:

1. **Загрузка данных**: 1m свечи за 30 дней для всех 10 тикеров через `get_all_candles()` (с автопагинацией)
2. **Ресемплирование**: 1m → 30m и 1H
3. **Симуляция**: проход по каждой 30m свече хронологически
   - Для каждого тикера: проверка открытых позиций (SL/TP/timeout), затем поиск новых сетапов
   - Позиция рассчитывается через `RiskManager.position_size()`
   - Баланс обновляется после каждой сделки
4. **Отчёт**: три блока вывода для каждого значения RR:
   - **BOARD 1**: общая статистика (баланс, P&L, return, win rate, avg win/loss)
   - **BOARD 2**: разбивка по тикерам (trades, W/L, win%, P&L)
   - **Trade log**: полная таблица сделок (тикер, направление, вход, выход, P&L, причина, бары)

Тестируемые значения RR: `2.0`.

Начальный баланс: `100 000 RUB`.

## Брокер (TinkoffBroker)

Обёртка над T-Bank gRPC API. Ключевые методы:

| Метод                  | Описание                                           |
|------------------------|----------------------------------------------------|
| `get_candles()`        | Получение свечей (один вызов, для live)             |
| `get_all_candles()`    | Получение свечей с пагинацией (для бэктеста)       |
| `place_limit_order()`  | Лимитный ордер на покупку/продажу                  |
| `place_stop_loss()`    | Стоп-ордер (стоп-лосс) на бирже                   |
| `place_take_profit()`  | Стоп-ордер (тейк-профит) на бирже                  |
| `cancel_order()`       | Отмена обычного ордера                             |
| `cancel_stop_order()`  | Отмена стоп-ордера                                 |
| `get_portfolio_balance()` | Баланс портфеля                                 |
| `sandbox_create_account()` | Создание аккаунта в песочнице                  |
| `sandbox_fund()`       | Пополнение виртуального баланса в песочнице         |

Эндпоинты:
- Sandbox: `sandbox-invest-public-api.tinkoff.ru:443`
- Live: `invest-public-api.tinkoff.ru:443`

## Хранение данных (MarketDataStorage)

SQLite база `data.db` с таблицей `candles`:

```sql
CREATE TABLE candles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    ticker TEXT NOT NULL,
    time TEXT NOT NULL,
    open REAL, high REAL, low REAL, close REAL,
    volume INTEGER
);
```

Метод `resample()` агрегирует 1-минутные свечи в более крупные таймфреймы (30m, 1H) через `pandas.DataFrame.resample()`.

## Архитектурный принцип

- Стратегии не зависят от брокера — они работают только с DataFrame и возвращают `Signal (BUY/SELL/HOLD)` или `ICTSetup`
- Все ордера проходят через `RiskManager` перед исполнением
- SL/TP выставляются на бирже, а не отслеживаются ботом — защита при падении процесса
